Openeuler系统Bioset病毒处理和系统加固

发表于 2022-12-13 更新于 2023-12-19 分类于 操作系统 ， 软件安装 ， 系统加固 本文字数： 2.1k 阅读时长 ≈ 8 分钟

系统环境

操作系统：OpenEuler22.03 LTS

服务器数量：N台

应用软件：wgcloud3.4.2（server+N个Agent）

问题发现

​ 公司内网的服务器状态监控一直是采用的wgcloud监控软件，偶然一天突然发现安装有agent的服务器不停地提示下线，但是远程过去后发现服务器并没有真正下线，期间公司运维主管人员反馈私有云监控平台发现其中两台机器的CPU和内存使用率不正常、较高，但通过ssh登录这两台机器后使用top命令发现使用率并不高，所以并没有理会。其中有几台服务器为开发测试机器，但是因为利用率不高，所以当时并未接到开发人员反馈问题。

​ 回过头来处理agent服务器掉线的问题，此时ssh登录agent服务器，想重启agent监控端应用，但提示发现无法启动，通过ll命令看到文件属性发生了变化：

​ 上图可以看出该执行文件的属性没有，此时想删除和重启都无法执行：

​ 通过报错关键字度娘得知，文件可能变成不可读不可写状态，通过lsattr和chattr结合chmod + x命令重新给该执行文件增加了可执行权限：

​ 此时，可以正常执行了，但是仅过了一会，该文件状态又变回了不可读状态，通过top命令也没有发现异常进程。与wgcloud作者讨论了半天也没有讨论出个结果。

​ 去看看其他agent服务器吧，看了以后才发现，之前看的服务器症状都是轻的：

❗ 症状摸排：

  1. 大部分主机出现N多ssh暴力尝试登录的记录（命令：lastb | less）；
  2. 部分主机出现自行进入紧急模式（Emergency mode），但输入密码，仍可执行部分命令；
  3. 部分主机出现ssh登录后，部分系统命令无法执行，提示not found，如ls等基础命令；
  4. 经抽样调查发现部分系统内应用执行文件也变成了不可读写状态，如docker相关执行文件；
  5. 部分主机上应用还在正常运行，现在不敢重启，怕重启后就启动不了了…

问题排查

​ 找一个linux水平比较高的哥们，希望帮忙分析一下问题，按他的要求提供了内核日志（dmesg）,他看了以后甩给我一个链接：https://www.cnblogs.com/shenyuanfeng/p/16031414.html ，反馈给我说可能是bioset病毒，但是还不确定。此时我去翻看dmesg日志文件，在日志中看到了相关关键字：

​ 我在度娘上找到了另一篇文章：(33条消息) linux 云服务器被暴破感染bioset_我啥都不会iii的博客-CSDN博客_bioset 采用文中的方法，使用busybox工具终于看到了bioset病毒进程：

​ 此时kill是无法结束的，他会自启，根本无法杀掉该进程。文中提供了杀死该进程的方法，这里不再赘述了，总结一下：

# 安装busybox
wget https://busybox.net/downloads/binaries/1.21.1/busybox-x86_64 --no-check-certificate
chmod +x busybox-x86_64
mv busybox-x86_64 /usr/local/bin/busybox
# busybox top 见上图
# kill病毒文件
busybox chattr -i /usr/bin/bioset && busybox rm -rf /usr/bin/bioset
# 删掉病毒进程，1247为上图的PID
kill -9 1247
# 此时病毒文件被删除了，此时busybox top再也不看到病毒进程了，也不会再自启了.....

✅ 其他服务器的处理：

删除病毒文件只能达到防止系统被进一步破坏，并不能起到彻底解决的作用。涉及的系统有OpenEuler和Centos系统。

病毒非常狡猾拦截篡改了系统动态库，导致我们使用top，ps等命令是查看不到病毒的进程的，这时就需要使用busybox工具协助我们排除。不过由于被篡改的文件较多，所以最后只能选择重装系统

系统加固

​ 通过这次事件，给了一个沉重的教训，针对该问题对系统加固有如下思考：

❗ 加固思考：

1.增加登录密码的复杂度（中毒的机器密码都是“1234qwer”）；
2.修改ssh默认端口号；
3.尽可能使用普通用户，关闭root的ssh登录；
4.如果服务器只与部分机器互通，可限定可访问该服务器的IP范围；

修改密码

可通过passwd命令来修改当前用户的密码，建议密码组成尽可能复杂，降低被破解的概率，大小写字母、数字、特殊符号，建议至少三种组合，密码长度>=8.

修改ssh配置

​ 通过修改ssh配置，变更ssh的默认端口号，关闭root的ssh登录权限，常规操作建议通过普通用户操作，需要操作系统命令的，普通用户可以通过命令前增加sudo来执行。

​ ！需要注意的是，修改ssh配置，需要保证selinux在disable或者permissive状态，否则修改后无法生效。

查询selinux状态：

# 查询selinux当前状态
sestatus

输出结果如下：

# SELinux启用状态(建议启用)
SELinux status:                 enabled
SELinuxfs mount:                /sys/fs/selinux
SELinux root directory:         /etc/selinux
Loaded policy name:             targeted
#SElinux当前模式
Current mode:                   permissive
Mode from config file:          permissive
Policy MLS status:              enabled
Policy deny_unknown status:     allowed
Max kernel policy version:      31

修改ssh配置命令如下:

# 修改ssh需要root权限,请确保使用root用户或在命令前增加sudo指令
vi /etc/ssh/sshd_config

# 修改其中的 Port 22段，移除Port 22前面的#，修改22为其他端口
# 为防止修改后无法ssh登录，可以在移除Port 22前面的#号后，换行增加一行Port 端口
# 修改完毕后，确认新端口可以正常ssh登录以后，再移除Port 22行
# 修改PermitRootLogin yes为no，该行代表是否允许root用户ssh登录。
# 确认修改完毕后按下ESC键，输入:wq保存退出

#	重启ssh服务
systemctl restart sshd
# 查看ssh服务状态
systemctl status sshd

限定IP访问访问范围

​ 特定应用部署的机器通过防火墙来限定能访问该机器指定端口的IP，可以降低被攻击的风险。举个例子，如果该服务器为Oracle数据库服务器，正常情况下，该服务器只接受应用服务器通过1521端口来访问该服务器。其他IP的访问都可以被拒绝。

​ 具体操作如下：

# OpenEuler系统使用的是Firewalld防火墙，我们以此举例

# 1.前提：确保firewalld服务正常运行

# 查看防火墙状态
systemctl status firewalld
# 开启防火墙
systemctl start firewalld
# 开机启动
systemctl enable firewalld

# 2.关闭允许公开访问的端口
# 查询打开的端口
firewall-cmd --zone=public --list-ports
# 如果输出结果为空，则跳过下面的关闭步骤

# 关闭端口9001(这里是举例，端口号请修改为需要关闭的端口号)
firewall-cmd --zone=public --remove-port=9001/tcp --permanent
# 重新载入一下防火墙设置，使设置生效
firewall-cmd --reload

# 3.加入限制访问规则
# 允许ip192.168.1.68访问本机的1521端口
# 如果允许多个IP访问，可以分次执行该命令
# 添加指定IP段，可以将命令的IP地址修改为“192.168.1.1/24”,请根据实际情况调整。
firewall-cmd --permanent --add-rich-rule="rule family="ipv4" source address="192.168.1.68" port protocol="tcp" port="1521" accept"
# 重新载入一下防火墙设置，使设置生效
firewall-cmd --reload
# 查看已设置规则
firewall-cmd --zone=public --list-rich-rules

可参考资料：Linux防火墙firewall只允许特定IP访问

其他问题

Q1.普通用户执行sudo命令提示不在sudoer范围内。

A1:修改”/etc/sudoers”配置文件，增加普通用户的sudo权限.

其中devtool用户为普通用户，如果有其他用户，复制行，将用户名修改为指定用户名即可，保存后即时生效。
